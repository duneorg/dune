# Skill: Dune Auth

Public (non-admin) user authentication. Two modes — pick one in `site.yaml`. Both produce the same `ctx.state.user` shape; all middleware works identically regardless of mode.

---

## Mode 1: `dune` — built-in session stack

Dune owns the session lifecycle. Credential verification is delegated to OAuth providers or magic link. **No passwords.**

### Minimal config (OAuth only, no server-side user store)

```yaml
# site.yaml
auth:
  mode: dune
  userStore: session       # OAuth claims encoded into cookie — no file or DB needed
  session:
    secret: "${SESSION_SECRET}"
    ttl: 7d
  providers:
    github:
      clientId: "${GITHUB_CLIENT_ID}"
      clientSecret: "${GITHUB_CLIENT_SECRET}"
```

### With persistent user records + magic link (no DB required)

```yaml
auth:
  mode: dune
  userStore: local          # flat files at data/users/ + Deno KV index
  providers:
    github:
      clientId: "${GITHUB_CLIENT_ID}"
      clientSecret: "${GITHUB_CLIENT_SECRET}"
    magicLink:
      enabled: true         # passwordless — requires email config
```

### With pre-created local users (no OAuth)

```yaml
auth:
  mode: dune
  userStore: local
  providers:
    localUsers:
      enabled: true         # users pre-created by admin in data/users/
    magicLink:
      enabled: true
```

`localUsers` is not a registration flow. Users are created by the admin (via `/admin/users/` or by editing files). Self-registration goes through an OAuth provider.

### Routes generated by mode 1

```
GET  /auth/login               → provider selection (or direct redirect if one provider)
GET  /auth/github              → begin OAuth flow
GET  /auth/github/callback     → complete OAuth, create session, redirect
GET  /auth/magic               → accept token from email link, create session
POST /auth/magic/send          → generate + send magic link email
POST /auth/logout              → destroy session
```

## Mode 2: `external-jwt` — validate and trust

Dune validates a JWT on each request, populates `ctx.state.user` from the claims, and does nothing else. No sessions, no OAuth flow, no user storage. The external service (Clerk, Auth0, Supabase Auth) owns everything.

```yaml
# site.yaml — Clerk example
auth:
  mode: external-jwt
  jwt:
    issuer: "https://your-app.clerk.accounts.dev"
    audience: "your-app"
    jwksUrl: "https://your-app.clerk.accounts.dev/.well-known/jwks.json"
    userClaims:             # map JWT claims → ctx.state.user fields
      id: sub
      email: email
      name: name
```

No routes generated. No sessions. No user records in Dune.

## `ctx.state.user` shape

Both modes populate the same shape. Always check for null before accessing fields.

```ts
interface PublicUser {
  id: string;
  email: string;
  name?: string;
  avatarUrl?: string;
  provider: string;       // "github" | "google" | "magic-link" | "clerk" | ...
  roles?: string[];       // group memberships — populated from JWT claims (external-jwt)
                          // or from authz tuples (local/db userStore)
  createdAt?: Date;       // absent in external-jwt mode
}

// ctx.state.user is null on unauthenticated requests — never undefined
```

## Protecting routes

### Redirect unauthenticated users

```ts
// routes/_middleware.ts  (or any nested _middleware.ts)
import type { FreshContext } from "fresh";

export async function handler(req: Request, ctx: FreshContext) {
  if (!ctx.state.user) {
    return Response.redirect(new URL("/auth/login", req.url));
  }
  return ctx.next();
}
```

`ctx.state.user` is populated globally on every request — no opt-in middleware required to get the value. The middleware above only adds the redirect gate.

### Require a specific role

```ts
import { authz } from "@/auth/authz.ts";

export async function handler(req: Request, ctx: FreshContext) {
  if (!ctx.state.user) {
    return Response.redirect(new URL("/auth/login", req.url));
  }
  const allowed = await authz.check({
    who: { type: "user", id: ctx.state.user.id },
    canThey: "access",
    onWhat: { type: "group", id: "member" },
  });
  if (!allowed) return new Response(null, { status: 403 });
  return ctx.next();
}
```

For content gating via frontmatter `roles:`, this check runs automatically in the page middleware — no manual wiring needed. See **dune-authz** skill.

## `userStore` tiers (mode: dune only)

| Value | User records | When to use |
|-------|-------------|-------------|
| `session` | None — OAuth claims encoded in cookie | OAuth-only, no need to identify users across requests beyond the session |
| `local` | `data/users/*.json` + Deno KV index | Default. No DB required. Supports localUsers, magicLink, OAuth. |
| `db` | DB table | Opt-in. Requires `db-schema-layer`. Use at scale or when admin UI user management is needed. |

`userStore: session` does not support `localUsers` — there's nowhere to store pre-created users. It only works with OAuth providers.

---

## Gotchas

**No passwords.** Password auth is explicitly out of scope. Magic link covers the passwordless use case. If a user needs to log in without OAuth, use `localUsers` + `magicLink`.

**`localUsers` always implies `userStore: local`.** You cannot use `localUsers` with `userStore: session` — users must exist somewhere before they can authenticate.

**`ctx.state.user` is `null`, not `undefined`.** Always check `if (!ctx.state.user)`, not `if (ctx.state.user === undefined)`.

**`external-jwt` eliminates user storage entirely.** Dune doesn't own user records, roles, or sessions in this mode. `ctx.state.user.roles` comes from JWT claims, not from `authz` tuples. Do not call `authz.addMember()` or `authz.allow()` expecting them to affect JWT claims.

**Session secret must be set.** `SESSION_SECRET` must be a strong random value (32+ bytes). Omitting it or using a weak value is a security issue. Generate with: `openssl rand -base64 32`.

**Magic link rate limiting.** `POST /auth/magic/send` has a fixed-window rate limit owned by Dune. If using an external provider (Clerk etc.) for magic links, rate limiting is that provider's concern.
